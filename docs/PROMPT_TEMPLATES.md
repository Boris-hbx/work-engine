## 二、优秀 Prompt 案例分析

### 案例 1: 拖拽功能 (评分: 9/10)

**原始 Prompt:**
```
ok，那现在可以解决todo里面事件拖拽功能了吧。就是鼠标单击某条事件，
点住不放，可以将其拖拽到其它象限，你是资深产品经理，我想你一定知道
我想要的是什么，请帮我实现出来。
```

**优点:**
- 明确了交互方式（单击、点住、拖拽）
- 指明了目标（拖到其他象限）
- 给予 Agent 角色定位（资深产品经理）

**改进建议:**
```
【功能名称】: Todo 拖拽分类

【功能描述】:
实现 Todo 事项的拖拽功能，可将任务拖到不同象限

【验收标准】:
1. 鼠标单击事项并按住，进入拖拽状态
2. 拖拽时显示视觉反馈（如半透明、阴影）
3. 拖到目标象限时高亮显示
4. 松开后任务移动到新象限
5. 数据自动保存到 todos.json

【技术约束】:
- 使用 HTML5 Drag & Drop API
- 兼容触屏设备
```

### 案例 2: 动画效果 (评分: 8/10)

**原始 Prompt:**
```
标签放大1.2倍就行了。另外看我截图，红框这里，我希望弄个动态效果，
比如有个红色的小点点，带着一个渐变颜色的尾巴，沿着这个框子顺时针游走，
反复循环。也可以是类似的效果。就是"My task"的这个框子。
```

**优点:**
- 具体的数值（1.2倍）
- 清晰的视觉描述（红色小点、渐变尾巴、顺时针）
- 参考截图辅助说明

**改进建议:**
```
【功能名称】: 边框光晕动画

【功能描述】:
在 "My task" 框子边缘添加动态光点效果

【具体要求】:
- 标签放大 1.2 倍
- 红色小点（约 8px）沿边框顺时针移动
- 带渐变色尾巴（长度约 50px）
- 动画周期: 5秒一圈
- 无限循环

【参考截图】: PrtSc/mytask_frame.png
```

### 案例 3: 游戏开发 (评分: 9/10)

**原始 Prompt:**
```
你是一名资深前端工程 Agent，擅长用 HTML + Canvas + 原生 JavaScript
构建轻量、可运行的网页小游戏。你的输出必须一次性可运行，不依赖任何外部资源。

🎯 任务目标
自动生成一个最小可玩版本（MVP）的 Xonix / Qix-like 网页小游戏

🧱 技术与输出约束
- 输出一个完整的 index.html
- 使用：HTML、<canvas>、原生 JavaScript
- ❌ 不使用任何框架、库、CDN
- ❌ 不拆分文件
- 打开 HTML 即可运行
```

**优点:**
- 明确角色定位
- MVP 思维，不过度设计
- 清晰的技术约束
- 使用 emoji 增强可读性

**评价:** 这是一个非常好的 Prompt 结构，适合复杂功能开发

### 案例 4: 问题修复 (评分: 7/10)

**原始 Prompt:**
```
在F12状态下，"重要且紧急"基本上不能拖拽，我成功了一次。其他可以。
```

**优点:**
- 描述了具体条件（F12状态）
- 说明了成功率（基本不能，成功一次）

**改进建议:**
```
【问题描述】: 拖拽功能在 F12 状态下不稳定

【复现步骤】:
1. 打开 F12 开发者工具
2. 尝试拖拽"重要且紧急"象限的任务
3. 重复 10 次

【实际结果】: 仅成功 1 次，其他 9 次无法拖动

【期望结果】: 100% 可拖动

【对比测试】: 其他象限可以正常拖拽
```

### 案例 5: 详细规格 (评分: 10/10)

**原始 Prompt:**
```
任务：
请为我生成一个可运行的网页端单人塔防游戏原型，同时生成一份塔防游戏设计文档

1. 技术选型：
   - 使用 HTML5 + CSS + JavaScript，建议使用 Canvas 或 Phaser.js
   - 代码结构模块化，便于未来扩展多人模式、移动端适配

2. 游戏基本功能：
   - 单人玩家可以放置防御塔
   - 敌人沿固定路径移动，防御塔自动攻击
   - 游戏分波次，每波敌人数量递增
   - 显示分数、击杀数和波次的简单 UI

3. 可扩展性要求：
   - 团队元素预留接口
   - 多人模式预留接口
   - 移动端预留

4. 输出要求：
   - 提供完整 HTML、CSS、JS 文件
   - 代码包含注释
   - 包含简单示例
```

**优点:**
- 结构化的需求描述
- 分层次列出要求
- 考虑了扩展性
- 明确的输出格式

**评价:** 这是最佳实践的典范，适合复杂项目开发

## 三、Prompt 编写检查清单

在提交 Prompt 前，检查以下要点：

- [ ] 功能描述是否清晰？
- [ ] 验收标准是否可量化？
- [ ] 技术约束是否明确？
- [ ] 是否提供了参考截图？
- [ ] 是否指定了影响范围？
- [ ] 是否考虑了边界情况？
- [ ] 是否兼容移动端/深色模式？

## 四、常见错误示例

| 错误写法 | 问题 | 正确写法 |
|---------|------|---------|
| "加个搜索功能" | 太模糊 | "添加 Todo 搜索，支持实时过滤，按 S 激活" |
| "优化一下" | 无目标 | "加载时间从 2s 优化到 1s 以内" |
| "修复 bug" | 无复现步骤 | "修复拖拽 bug，复现：1. 刷新 2. 拖拽..." |
| "改成好看点" | 主观无标准 | "背景色改为 #E3F2FD，字体放大到 16px" |
| "你看着办" | 无验收标准 | 提供明确的验收标准列表 |

## 五、效率提升技巧

1. **第一次就写清楚** - 5 分钟写 Prompt，省 30 分钟返工
2. **善用截图** - 一图胜千言，保存到 `PrtSc/` 目录
3. **小步快跑** - 大任务拆成小任务，逐个验证
4. **即时反馈** - 完成就测试，问题早发现
5. **模板复用** - 使用本文档的模板，保持一致性
